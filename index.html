Qu'est-ce que WebAssembly ?

Le site Web officiel de WebAssembly (https://webassembly.org) propose cette définition :

WebAssembly (abbreviated Wasm) is a binary instruction format for a stack-based virtual machine. Wasm is designed as a portable target for compilation of high-level languages like C/C++/Rust, enabling deployment on the web for client and server applications.

WebAssembly est un langage pouvant être exécuté dans les navigateurs, il fournit des gains majeurs en performance, il est conçu principalement pour être une cible de compilation efficace pour les langages de bas niveau tels que C, C ++ et Rust. Il fournit un moyen d'exécuter un code écrit dans divers langages sur le web à une vitesse proche du natif.

Objectifs de WebAssembly

Efficace, Rapide et Portable

La machine de pile WebAssembly est conçue pour être encodée dans un format binaire efficace en termes de taille et de temps de chargement. WebAssembly vise à s'exécuter à la vitesse native en tirant parti des capacités matérielles communes disponibles sur une large gamme de plates-formes.

Lisible et Debuggable

WebAssembly est conçu pour être imprimé dans un format textuel pour le débogage, le test, l'expérimentation, l'optimisation, l'apprentissage, l'enseignement et l'écriture de programmes à la main. Le format textuel sera utilisé lors de la visualisation de la source des modules Wasm sur le Web.

Sécurisé

WebAssembly décrit un environnement d'exécution sécurisé en mémoire, qui peut même être implémenté dans des machines virtuelles JavaScript existantes. Lorsqu'il est intégré au Web, WebAssembly applique les politiques de sécurité de même origine et autorisations du navigateur.

Partie de Web

WebAssembly est conçu pour maintenir la nature sans version, testée et rétrocompatible du Web. Les modules WebAssembly pourront appeler dans et hors du contexte JavaScript et accéder aux fonctionnalités du navigateur via les mêmes API Web accessibles à partir de JavaScript. WebAssembly prend également en charge les intégrations non Web.

Cas d'utilisation

Le site officiel de WebAssembly contient une longue liste de cas d'utilisation potentiels. Techniquement, certains de ces cas d’utilisation sont réalisables avec juste de JavaScript, CSS et HTML, mais l’utlisation de WebAssembly peut offrir des gains majeurs en terme de performances, en fait l’utilisation d’un fichier binaire au lieu d'un seul fichier JavaScript peut réduire considérablement la taille du bundle ainsi que l'instanciation du module WebAssembly au chargement de la page accélère l'exécution du code.

Nous ne pouvons pas couvrir toute la liste des cas d’utilisation, mais voici quelques cas d’utilisation principaux qui représentes des améliorations significatives de la capacité de web :

  • Edition d'images et des vidéos
  • Jeux
  • Reconnaissance d'image
  • Applications musicales
  • Réalité Virtuelle et Réalité Augmentée
  • Visualisation et simulation scientifiques

JavaScript et WebAssembly

WebAssembly et JavaScript sont deux langages différents, et l’objectif principal de WebAssembly n’est pas de remplacer JavaScript mais de le compléter. WebAssembly est un langage bas niveau, comme l’Assembleur, il a un format binaire qui tourne avec des performances proches du natif et il est considéré comme une cible de compilation des langages de haut niveau comme C, C++ et Rust afin de pouvoir les tourner sur le web, et JavaScript est un langage haut niveau, flexible et suffisamment expressif pour écrire des applications web. Il a beaucoup d'avantages, à titre d’exemple, il est dynamiquement typé, il ne nécessite aucune étape de compilation, et il a un grand écosystème qui lui fournit de puissants frameworks, outils et APIs.

Les deux langages travaillent ensemble afin de permettre aux développeurs web de prendre avantage des avantages de chacun d’eux. Les deux différents types de code peuvent s'appeler si nécessaire, en fait l'API JavaScript de WebAssembly rend l'interopérabilité relativement simple entre les deux technologies, l’API enveloppe le code exporté avec des fonctions JavaScript qui peuvent être appelées normalement et le code WebAssembly peut importer et appeler normalement de manière synchrone les fonctions JavaScript.

Concepts du langage

La spécification de base indique que le code de WebAssembly est un langage de programmation de type assembleur de bas niveau. La spécification définit la structure, l'exécution et la validation de ce langage ainsi que les détails des formats binaire « wasm » et texte « wat ». Le langage WebAssembly est structuré autour de plusieurs concepts qui se retrouvent totalement dans l'API WebAssembly JavaScript.Ces concepts sont nécessaires pour comprendre le fonctionnement de WebAssembly dans les navigateur :

  • Valeurs, ou plutôt types de valeurs fournis par WebAssembly.
  • Table, qui est un tableau de valeurs d'un type d'élément particulier (tels que les références de fonction) qui ne peut pas être stocké de manière brute en mémoire pour des raisons de sécurité et de portabilité.
  • Mémoire, un ArrayBuffer redimensionnable qui contient un tableau d'octets contiguë accessible en lecture/écriture par les instructions bas niveau d'accès mémoire du WebAssembly.
  • Instance, un module associé avec tous les états qu'il utilise à l'exécution à savoir la mémoire, la table et un ensemble de données importées. Une instance est comme un module ES2015 qui a été chargée dans un contexte global avec un ensemble d'imports.
  • Module, représente un binaire WebAssembly qui a été compilé en code exécutable par le navigateur. Un module est sans état et peut donc être explicitement mis en cache dans IndexedDB ou partagé entre le contexte fenêtre et les workers. Un module déclare des imports et exports au même titre qu'un module ES2015.

Phases sémantiques

La spécification principale décrit les différentes phases qu'un module WebAssembly subit lorsqu'il est utilisé dans un environnement tel qu'un navigateur. Conceptuellement, la sémantique de WebAssembly est divisée en trois phases, dont la troisième phase est divisée aussi en deux sous phases :

  • Décodage : le format binaire est converti en module.
  • Validation : le module décodé subit des vérifications de validation (telle que la vérification de type) pour s'assurer que le module est bien formé et sécurisé.
  • Instanciation (partie 1 de l'Exécution) : une instance de module, qui est la représentation dynamique du module, est instanciée en initialisant les mémoires et les tables et appelle la fonction start () du module.
  • Invocation (partie 2 de l'Exécution) : les fonctions exportées sont appelées à partir de l'instance de module.

Formats de WebAssembly

WebAssembly a deux formats, un format binaire et un format texte. Les deux formats correspondent à une structure commune sous la forme d'une syntaxe abstraite. Pour mieux comprendre la syntaxe abstraite, elle peut être expliquée dans le contexte d'un arbre de syntaxe abstraite (AST). Un AST est une représentation arborescente de la structure du code source d'un langage de programmation.

Format binaire (.wasm)

Le format binaire de WebAssembly est très efficace, il permet un décodage rapide, une petite taille de fichier et une utilisation réduite de la mémoire. La représentation de fichier du format binaire est un fichier « .wasm », qui sera la sortie de compilation d'Emscripten. Les concepts Valeurs, Types et Instructions sont couverts dans le contexte de l'encodage. Par exemple, selon la spécification, les types Entier sont codés à l'aide du codage entier de longueur variable LEB128, dans une variante non signée ou signée. Ce sont des détails importants à savoir si vous souhaitez développer des outils pour WebAssembly, mais pas nécessaires si vous prévoyez simplement de les utiliser sur votre site Web.

Format texte (.wat)

De la même manière que les langages assembleur physique, le format binaire du WebAssembly a une représentation textuelle . Le format textuel des modules WebAssembly est un rendu de leur syntaxe abstraite en S-expressions qui sont des notations pour les données de liste imbriquées (arborescentes). Essentiellement, ils fournissent un moyen simple et élégant de représenter des données basées sur une liste sous forme textuelle.

Le format texte permet la représentation d'un module WebAssembly binaire sous forme textuelle. Cela a des implications profondes en ce qui concerne la facilité de développement et de débogage. Avoir une représentation textuelle d'un module WebAssembly permet aux développeurs de visualiser la source d'un module chargé dans un navigateur. Le site officiel décrit les cas d'utilisation qui ont conduit à la conception du format texte :

  • Afficher la source sur un module WebAssembly, s'intégrant ainsi au Web (où chaque source peut être consultée) de manière naturelle.
  • Présentation dans les outils de développement de navigateur lorsque les cartes sources ne sont pas présentes.
  • Écrire le code WebAssembly directement pour des raisons pédagogiques, expérimentales, débogage, optimisation et test de la spécification elle-même.

    • Le dernier élément de la liste indique que le format de texte n'est pas destiné à être écrit à la main au cours du développement normal, mais plutôt généré à partir d'un outil comme Emscripten. Vous ne verrez ou ne manipulerez probablement aucun fichier « .wat » lorsque vous générez des modules, mais vous pouvez les visualiser dans un contexte de débogage.

Langages supportés

C / C++

La prise en charge de C et C ++ est également intégrée à Emscripten, donc en plus de simplifier le processus de compilation, Emscripten nous permet de tirer parti des capacités complètes de WebAssembly. L’outil permet de compiler C/C++ dans un module WebAssembly, de générer un code JavaScript qui va permettre le chargement et l’exécution du module ainsi que la création d’un fichier HTML qui va afficher les résultats d’exécution du code compilé.

Emscripten compile le code à travers la chaîne de compilation « clang + LLVM », ensuite il transforme le résultat compilé en binaire « .wasm ». Emscripten crée le document HTML et le code JavaScript nécessaire, car WebAssembly ne peut pour l’instant pas accéder directement au DOM, il appelle JavaScript avec des données du type primitif entier ou flottant, ainsi pour accéder à toute API Web, WebAssembly a besoin d'appeler du JavaScript qui va effectuer par la suite l'appel à l'API Web.

Emscripten implémente des bibliothèques C/C++ comme SDL, OpenGL, OpenAL, ces bibliothèques sont implémentées sous forme d'API Web et donc chacune d'entre elles requiert un peu de code JavaScript généré pour relier WebAssembly à l'API Web sous-jacente, ainsi que le code JavaScript implémente les fonctionnalités de chaque librairie utilisée par le C/C++. Le code JavaScript contient aussi la logique pour appeler l'API JavaScript WebAssembly pour chercher, charger et exécuter le fichier « .wasm ».

Le document HTML généré charge le fichier JavaScript contenant le code JavaScript généré et écrit un « stdout » dans un « TextArea », et si l'application utilise OpenGL, le fichier HTML contient aussi un élément qui est utilisé comme cible de rendu. En fait, il est simple de modifier la sortie Emscripten pour en faire l'application web que nous souhaitons.

Rust

C et C ++ étaient censés être les principaux langages utilisés pour WebAssembly, mais Rust est un substitut parfaitement adapté. Rust est un langage de programmation de systèmes qui est syntaxiquement similaire au C ++. Il a été conçu en pensant à la sécurité de la mémoire, mais conserve les avantages de performances du C et du C ++. La version actuelle du compilateur de Rust peut générer des fichiers « .wasm » à partir du code source de Rust.

Autres langages

Techniquement il est possible de transpiler un langage en C ou C++ et par conséquent le compiler dans un module WebAssembly, mais le succès de la compilation dépend de la sortie du transcripteur, souvent nous devrons apporter des modifications au code pour le faire fonctionner.

Sinon, divers outils existent pour permettre l'utilisation de WebAssembly avec certains des autres langages de programmation, bien qu'ils soient principalement expérimentaux :

  • Blazor pour C#
  • TeaVM ou Bytecoder pour Java
  • TeaVM pour Kotlin
  • AssemblyScript pour TypeScript
  • WebIDL pour Haxe
